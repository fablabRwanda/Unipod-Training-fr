Contrôle de Version

Qu'est-ce que Git ?

Git est un système de contrôle de version distribué qui suit les modifications de vos fichiers de projet au fil du temps, permettant la collaboration, l'historique, le branchement et l'expérimentation sécurisée.

Qu'est-ce que GitHub ?

GitHub est une plateforme cloud pour héberger des dépôts Git. Il ajoute des fonctionnalités de collaboration (problèmes, demandes de tirage), le contrôle d'accès et des outils de déploiement comme GitHub Pages pour les sites web statiques.

Installer et Configurer Git
  • Windows: Download and install Git for Windows. During setup, accept defaults (includes Git Bash).
  • macOS:
    xcode-select --install (installs Command Line Tools with Git)
    or install Homebrew first from brew.sh, then brew install git
  • Linux (Debian/Ubuntu):
    sudo apt update && sudo apt install -y git
  • Linux (Fedora/CentOS/RHEL):
    sudo dnf install -y git (Fedora) or sudo yum install -y git (CentOS/RHEL)
  • Verify: git --version
  • Identify yourself:
    git config --global user.name "Your Name"
    git config --global user.email "you@example.com"
Create SSH Keys and add to GitHub
  1. Generate a key: ssh-keygen -t ed25519 -C "you@example.com" (press Enter to accept defaults, set a passphrase if you want).
  2. Start the agent and add your key:
    eval "$(ssh-agent -s)"
    ssh-add ~/.ssh/id_ed25519
  3. Copy the public key:
    pbcopy < ~/.ssh/id_ed25519.pub (macOS)
  4. Add it in GitHub → Settings → SSH and GPG keys → New SSH key.
Basic Git Workflow
  1. Clone: git clone git@github.com:YOUR_USER/YOUR_REPO.git
  2. Create a branch: git switch -c feature/my-change
  3. Stage and commit: git add -A && git commit -m "Describe your change"
  4. Push: git push -u origin feature/my-change
  5. Open a Pull Request on GitHub and merge.
Host a Website with GitHub Pages
  1. Create a repository named YOUR_USER.github.io.
  2. Add your static site (HTML/CSS/JS) at the repo root.
  3. Commit and push to main.
  4. Your site will be available at https://YOUR_USER.github.io within a few minutes.

  • Option 1 (user site): push your site to YOUR_USER.github.io.
  • Option 2 (project site): enable Pages in Repository Settings → Pages → set Source to main (or gh-pages) and folder to / or /docs.

Core Concepts (Learn these terms)
  • Repository (repo): a project folder tracked by Git.
  • Commit: a snapshot of your changes with a message.
  • Branch: a line of development (e.g., main, feature/home-hero).
  • Merge: bring changes from one branch into another.
  • Remote: a copy hosted elsewhere (e.g., GitHub origin).
  • HEAD: where you currently are in history/branch.

Hosting Workflow: Static Website with Git and GitHub Pages
  1. Plan: Decide site structure (e.g., index.html, labs/*.html, course-content/*.html).
  2. Create a repo:
    • User site: name repo YOUR_USER.github.io.
    • Project site: any name; Pages will serve from a branch/folder.
  3. Clone: git clone git@github.com:YOUR_USER/YOUR_REPO.git
  4. Develop locally: edit HTML/CSS/JS, open index.html in the browser to preview (or use VS Code Live Server).
  5. Branch: git switch -c feature/site-structure for non‑trivial work.
  6. Commit: git add -A then git commit -m "Add nav and content sections"
  7. Push: git push -u origin feature/site-structure
  8. Pull Request: open a PR on GitHub; get review; Merge.
  9. Deploy with GitHub Pages:
    • User site: push to main of YOUR_USER.github.io. URL is https://YOUR_USER.github.io.
    • Project site: in Settings → Pages, set Source to Main (or gh-pages) and folder / (or /docs). URL is https://YOUR_USER.github.io/REPO_NAME/.
  10. Iterate: repeat branch → commit → PR → merge to keep history clean.
Tip: For project sites, a common pattern is to publish from the gh-pages branch or the docs/ folder on main.

Best Practices for Étudiants
  • Write meaningful commit messages (why + what).
  • Use feature branches for bigger changes; keep main stable.
  • Keep paths relative (e.g., ./images/pic.png) to avoid broken links on Pages.
  • Organize assets in folders: images/, assets/, css/, js/.
  • Preview locally before pushing; use your browser dev tools for errors.
  • Add a README describing the project and how to build/preview.
Troubleshooting
  • 403 when pushing: ensure you have access. Use SSH remote git@github.com:ORG/REPO.git or HTTPS with a Personal Access Token (repo scope).
  • 404 on GitHub Pages: Pages may take a few minutes. Verify Settings → Pages is enabled and the selected branch/folder has index.html.
  • CSS/JS not loading: check relative paths and case sensitivity (GitHub Pages is case-sensitive).
  • Wrong site URL: user sites use YOUR_USER.github.io; project sites add /REPO_NAME/ path.
  • Custom domain: add CNAME in Settings → Pages and configure DNS (A/ALIAS to GitHub, plus CNAME record).
Quick Reference
git clone git@github.com:YOUR_USER/YOUR_REPO.git
git switch -c feature/my-change
git add -A
git commit -m "Explain your change"
git push -u origin feature/my-change
git switch main && git pull
git merge --no-ff feature/my-change
git push origin main
  • git clone: downloads the remote repository to your computer.
  • git switch -c: creates and moves to a new branch for isolated work.
  • git add -A: stages all changes (new, modified, deleted files) for commit.
  • git commit -m: saves a snapshot with a message explaining the change.
  • git push -u origin BRANCH: uploads your branch to GitHub and tracks it.
  • git switch main && git pull: return to main and update it from the remote.
  • git merge --no-ff: merges the feature branch, preserving a merge commit for history clarity.
  • git push origin main: publishes the merged changes to GitHub (triggers Pages if enabled).
Glossary
  • Fork: your own copy of someone else’s repo on GitHub.
  • Pull Request (PR): propose changes from a branch/fork into another branch.
  • Rebase: replay your commits on top of another branch.
  • Remote tracking branch: local branch linked to a remote branch (e.g., origin/main).
  • CI: automated checks (build/tests) that run on pushes/PRs.
